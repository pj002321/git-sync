객체 생성 로직을 캡슐화하며, 객체 생성을 전담하는 패턴

왜 쓰는지?
- 객체 생성 코드를 분리해 클라이언트가 구체 클래스에 의존하지 않도록 하기 위해 사용.
언제 쓰는지?
- 종류가 많고 생성 규칙이 복잡한 객체를 만들 때 팩토리 패턴을 사용하여 생성 책임을 분리하고, 타입 추가나 변경 시 기존 코드 수정 없이 확장 가능.
장점
- 생성 로직이 한곳에 모여 유지보수가 쉽고, 새로운 타입 추가 시 기존 코드 수정이 최소화된다.

단점
- 클래스 수가 증가하고 구조가 복잡해질 수 있다.

구조
- 객체 생성을 담당하는 Factory와, Factory가 생성하는 공통 인터페이스 또는 추상 클래스로 구성된다.

```
            +------------------+
            |   Product        |  ← 공통 인터페이스 / 추상 클래스
            |------------------|
            | + Use()          |
            +------------------+
                     ▲
        --------------------------------
        |                              |
+------------------+        +------------------+
| ConcreteProductA |        | ConcreteProductB |
|------------------|        |------------------|
| + Use()          |        | + Use()          |
+------------------+        +------------------+


            +------------------+
            |   Factory        |  ← 객체 생성 담당
            |------------------|
            | + Create()       |
            +------------------+

```
